# Re-implementation: Fix Review Issues for Unit rust-web-search-brave-to-tavily

The original migration plan is at: workspaces/motosan-dev--motosan-agent-tool/.migration-plan-rust-web-search-brave-to-tavily.md

## Review Feedback

The following issues were found during code review. Fix these specific issues. Do NOT re-do the entire migration — only address the problems below.

### Issue 1 [MAJOR]: Cargo.toml

**Problem:** The `web_search_tavily` feature flag is non-functional. `src/tools/mod.rs` gates the entire `web_search` module on `#[cfg(feature = "web_search")]` only. There is no `#[cfg(feature = "web_search_tavily")]` attribute anywhere in the codebase, so enabling `web_search_tavily` without `web_search` compiles nothing, and enabling `web_search` always compiles the Tavily code regardless of whether `web_search_tavily` is set. The feature flag is advertised in `all_tools` but is purely dead — it neither gates code nor adds any capability on its own.
**Suggestion:** Either (a) add `#[cfg(any(feature = "web_search", feature = "web_search_tavily"))]` to `mod.rs` and gate the Tavily structs/methods inside `web_search.rs` with `#[cfg(feature = "web_search_tavily")]`, making it a true opt-in; or (b) remove the separate `web_search_tavily` feature and fold Tavily support directly into `web_search` (no separate flag), which is simpler given they share the same file and same `reqwest` dep.

### Issue 2 [MAJOR]: src/tools/web_search.rs

**Problem:** The Tavily API call passes the API key in the JSON request body (`"api_key": api_key`). The current Tavily REST API requires Bearer-token authentication via the `Authorization` header (`Authorization: Bearer <key>`), not a body field. Sending the key in the body will result in 401 Unauthorized responses for all Tavily requests.
**Suggestion:** Remove `"api_key"` from the `json!({})` body and add `.header("Authorization", format!("Bearer {}", api_key))` to the `self.http.post(...)` builder chain. The request body should only contain `query` and `max_results` (and any other optional Tavily parameters).

### Issue 3 [MINOR]: src/tools/web_search.rs

**Problem:** Provider-selection tests manipulate global `std::env` state without using a serialisation primitive (e.g., the `serial_test` crate). Since `cargo test` runs tests in parallel by default, concurrent tests that also read/write `BRAVE_API_KEY`, `TAVILY_API_KEY`, or `SEARCH_PROVIDER` can race and produce flaky results.
**Suggestion:** Add `serial_test` to `[dev-dependencies]` and annotate the three new env-var tests (and the existing `should_fail_without_api_key`) with `#[serial]` to serialise their execution.

### Issue 4 [MINOR]: src/tools/web_search.rs

**Problem:** `SEARCH_PROVIDER` matching is case-sensitive (`"brave"` / `"tavily"` only). A value like `"Brave"`, `"TAVILY"`, or `"Tavily"` silently falls through to the default provider-preference logic instead of honouring the intent.
**Suggestion:** Lowercase the preference string before matching: `match preference.as_deref().map(|s| s.to_ascii_lowercase()).as_deref() { ... }` — or use `eq_ignore_ascii_case` in the match arms.

### Issue 5 [MINOR]: src/tools/web_search.rs

**Problem:** When `SEARCH_PROVIDER` is explicitly set to `"tavily"` but `TAVILY_API_KEY` is absent, `select_provider` returns `None` and the caller emits a generic "No search API key is configured" error, giving no indication that a specific provider was requested but its key is missing.
**Suggestion:** In the `Some("brave")` and `Some("tavily")` arms, return a provider-specific error when the corresponding key is absent — e.g., `ToolResult::error("SEARCH_PROVIDER=tavily is set but TAVILY_API_KEY is not configured")` — rather than silently returning `None`.
